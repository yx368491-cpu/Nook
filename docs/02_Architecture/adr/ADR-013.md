# ADR-013 · 状态管理 Zustand (client) + TanStack Query v5 (server)

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

Nook 前端需要管理两种不同性质的数据：
- **Client state**：UI 状态（当前选中的会话、composer draft、modal 开关、语言选择）。
- **Server state**：消息列表、会话列表、朋友资料、反应计数 —— 这些数据是服务端权威的缓存镜像。

React 生态有多种状态管理方案，选择权需要匹配 Nook 的体量（≤ 20 好友 SPA）。

## Decision

- **Client state → Zustand**：轻量（< 2KB）、直观、不需要 Redux 的样板代码。
- **Server state → TanStack Query v5**：自动缓存、revalidation、stale-while-revalidate、devtools。

职责分离：
- `src/stores/`（Zustand）：`useChat`（active conv / draft / composer）、`useUI`（sidebar / modal / locale）、`useAuth`（current user / role）、`usePresence`（在线朋友 dict）。
- `src/lib/api/` + TanStack Query：消息 CRUD、会话列表、反应 toggle。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Zustand + TanStack Query（选中）** | 各司其职；体量匹配 | 两个库的学习成本 |
| **Redux Toolkit** | 生态丰富 | chat 体量不需要；样板代码多 |
| **仅 useState + useEffect** | 零依赖 | 缺乏缓存/revalidation；聊天应用是大忌 |
| **Jotai / Recoil** | 原子化状态 | 不如 Zustand + TQ 组合成熟 |

## Consequences

- `src/stores/` 和 `src/lib/api/` 必须严格分离。
- TanStack Query 管理所有来自 `/rest/v1/*` 的请求。
- Realtime 推送直接更新 query cache（无需额外 store）。

## 是否允许未来修改

**中可逆性**——可从 Zustand 迁移到 Jotai，但 TanStack Query 对于 server state 已是行业标准。
