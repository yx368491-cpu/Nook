# ADR-020 · 测试策略（Unit + Integration + E2E）

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

Nook 是私人项目（非企业级），测试策略需要在质量保障与开发速度之间取得平衡。同时需要支持 CI 流水线自动运行。

## Decision

**3 层测试策略**：

| 类型 | 工具 | 覆盖范围 | 频率 | 运行时间预期 |
|---|---|---|---|---|
| **Unit** | Vitest + Testing Library | 原子组件、hooks、store、pure functions | 每次 commit | < 30s |
| **Integration** | Vitest + Supabase local | API endpoint 调用（mock Supabase local DB） | PR CI | < 2min |
| **E2E** | Playwright | 关键业务流程（BF-01 ~ BF-15） | PR CI + main 部署前 | < 3min |

**目录结构**：



```
tests/
├── unit/
├── integration/
│   ├── api/
│   └── realtime/
├── e2e/
│   ├── auth.spec.ts
│   ├── chat.spec.ts
│   ├── invite.spec.ts
│   ├── admin.spec.ts
│   └── responsive.spec.ts
├── mocks/
│   ├── factories.ts
│   ├── handlers.ts
│   └── supabase.ts
├── fixtures/
└── utils/
```



## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **仅 Unit 测试** | 写最快 | 无法覆盖 RLS / 业务流程集成；上线风险高 |
| **Unit + Integration + E2E（选中）** | 质量兜底完整；CI 自动化 | 初期写测试耗时较多 |
| **TDD 全覆盖** | 最高质量 | 个人 MVP 阶段速度过慢 |

## Consequences

- E2E 覆盖 5 个 spec 文件：注册/邀请/聊天/管理/响应式。
- RLS 穿透测试（`scripts/test-rls.ts`）作为 Integration 测试的一部分。
- Vitest 配置 `setup.ts` + `tsconfig` 引用。

## 是否允许未来修改

**高可逆性**——可增减测试类型（如加 visual regression），但 CI 必须保持运行。
