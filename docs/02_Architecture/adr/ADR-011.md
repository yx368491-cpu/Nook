# ADR-011 · 实时通信 Supabase Realtime (WS + Presence)

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

Nook 需要实时消息推送（新消息、编辑、撤回、反应）+ typing 指示 + 在线状态。传统方案需要自建 WebSocket 服务器或引入专用实时平台。

## Decision

**Supabase Realtime**（WebSocket + Presence）。

- 200 并发连接免费（Nook ≤ 30 连接，永不爆）。
- Presence API 直接支持 typing + 在线状态（零自写 WS 协议）。
- `postgres_changes` 监听 messages / reactions 表的 INSERT/UPDATE/DELETE 事件。
- 5 个 Realtime channel 命名约定：
  - `conversation:<uuid>` — 消息 + 反应
  - `presence:<uuid>` — 在线 + typing
  - `user:<uuid>` — 个人通知
  - `admin:owner` — Owner 仅限邀请管理

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Supabase Realtime（选中）** | 零维护；含 Presence；REST 生态整合 | 单供应商依赖 |
| **Pusher** | 成熟；免费层可用 | 额外供应商；不整合 Postgres |
| **自建 WS（Node + ws）** | 完全控制 | 需要维护 WS 服务器；Presence 自实现 |
| **Ably** | 可靠；Push 通道 | 额外费用；配置复杂 |

## Consequences

- 5 个 Realtime channel 命名约定必须在客户端遵守。
- `postgres_changes` 的缺点：filter 精确度有限（按列值 filter，不支持复杂 query）。
- 断连重连由 `supabase-js` SDK 自动处理（< 3s）。

## 是否允许未来修改

**中可逆性**——可迁 Puhser / Ably / 自建 WS，但需重新实现 Presence 逻辑。
