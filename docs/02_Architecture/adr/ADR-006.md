# ADR-006 · 4 群 + 8 成员硬上限 = DB trigger + UI 双重

## Status

✅ **Accepted**

## Date

2026-06-27

## 修正历史

- 7/6 update（thinker 反馈）：T-01 改为 `WHERE kind='group'`（不含 1:1）

## Context

`../../01_Product/Nook-SPEC.md` § 1.8 规定 4 群硬上限、每群成员上限 8 人。实现层需要确保该约束不可绕过——前端 disabled 按钮可以被绕过（直接发 HTTP 请求），因此需要在 DB 层做权威约束。

## Decision

- DB trigger `trg_conversations_cap` BEFORE INSERT on conversations：仅统计 `kind='group'` 的行数 ≥ 4 时 `raise exception 'CONV_HARD_CAP'`。
- DB trigger `trg_conversation_members_cap` BEFORE INSERT on conversation_members：统计 active members（`left_at IS NULL`）≥ 8 时 `raise exception 'MEMBER_HARD_CAP'`。
- UI 端：创建群按钮 disabled + tooltip「已达 4 群上限」；invite 按钮 disabled + tooltip「该群已满 8 人」。
- 编辑窗口 trigger `trg_messages_edit_window` 属于同一决策家族（D-07）。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **仅 UI 层限制** | 实现快 | 易绕过（直接 POST） |
| **DB trigger + UI（选中）** | DB-authoritative；UI 体验好 | 额外 trigger；必须精确区分 group vs 1:1 |

## Consequences

- 两条 DB trigger 是硬防线。
- UI disabled 是体验层——即使用户绕过 UI，DB 也会拒绝。
- 1:1 conversation 不受 4 群上限影响（`kind='one_to_one'` 不计入）。

## 是否允许未来修改

**高可逆性**——任意时刻可改上限数值或取消 trigger。
