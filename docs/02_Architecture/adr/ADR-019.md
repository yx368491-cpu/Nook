# ADR-019 · 错误处理统一格式 + 映射

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

Nook 有 3 类端点（REST / Edge Function / Realtime），错误格式需要统一，以便前端统一处理（显示 toast、触发 fallback、上报 Sentry）。同时需要将 Supabase 原生错误映射为友好的应用层错误码。

## Decision

**统一错误格式**：



```typescript
interface ApiError {
  error: {
    code: ErrorCode;           // 机器可读枚举
    message: string;           // 人类可读（i18n key）
    details?: Record<string, unknown>;
    request_id?: string;       // Sentry 关联
  };
}
```



**4 类别错误码**：

- `E_AUTH_*`（认证/鉴权：401/403）。
- `E_VAL_*`（校验：422 — 字段、限制、窗口过期）。
- `E_RES_*`（资源：404/409/410 — 不存在、已满、过期）。
- `E_SYS_*`（系统：429/500/503 — 限流、内部错误、维护）。

**Supabase 错误映射**（`lib/api/errors.ts`）：

- `42501` → `E_AUTH_FORBIDDEN`。
- `P0001`（trigger exception）→ 按 message 内容映射为 `E_VAL_*` 或 `E_RES_*`。
- SQL 细节永不暴露给客户端。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **统一错误码（选中）** | 前端可 switch；Supabase 错误被隔离 | 需要维护映射表 |
| **直接暴露 Supabase 错误** | 零额外代码 | SQL 细节泄漏；前端难以统一处理 |
| **仅 HTTP status code** | 最简 | 缺少业务语义（如「4 群上限」vs「群满 8 人」） |

## Consequences

- `lib/api/errors.ts` 实现 `mapSupabaseError()` 映射函数。
- 所有 trigger exception 必须包含可区分的 message（如 `'CONV_HARD_CAP'`、`'EDIT_WINDOW_EXPIRED'`）。
- 前端 toast 组件使用 `error.code` 决定展现文案和严重程度。

## 是否允许未来修改

**高可逆性**——可增删 error code 分类，但格式一旦发布应保持向后兼容。
