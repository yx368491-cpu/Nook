# ADR-001 · SPEC = Single Source of Truth

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

在项目早期（Stage 6 Review）发现多份源文档（../../01_Product/Nook-PRODUCT.md、../Nook-ARCHITECTURE.md、../../01_Product/Nook-INTERVIEW-spec.md）之间存在多处局部矛盾（如 Web Push 在 PRODUCT 标 MUST、在 SPEC 标 ❌；i18n 在 ARCH 写 3 语、Spec 裁决双语）。开发团队难以确定哪份文档是最终权威。

## Decision

`../../01_Product/Nook-SPEC.md` 被正式任命为**唯一可信的需求来源（Single Source of Truth）**。

规则：
- 任何与 SPEC 矛盾的代码、组件实现、组件 .spec.md 文档 → 视为 bug，必须以 SPEC 为准。
- SPEC 不复制详细的 token 值 / 颜色 hex / architecture 选型——这些引用对应源文档。
- SPEC 变更必须通过版本号（v1.0 → v1.0.1 → v1.1）追加，不修改历史。
- 源文档（PRODUCT / ARCH 等）提供补充说明但不具备替代 SPEC 的权威。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **多份独立 spec + 校验脚本** | 每份文档可以独立演进 | 维护成本线性增长；小团队收益 < 成本 |
| **唯一 SoT + 自动生成子文档** | 源端一处修改、处处同步 | 需要自建 CI 流水线；初期成本 > 手工维护 |
| **产品文档驱动**（PRODUCT.md 为 SoT） | 产品经理熟悉 | 缺少技术实现细节，不适合直接指导开发 |

## Consequences

**积极**：
- 开发期有明确锚点，减少「我说的是 / 你说的是」语义漂移。
- 所有 v1.x 开发 / 测试 / 部署以 SPEC 为准。
- 评审时只需检查 SPEC 与实现的一致性。

**消极**：
- 需要主动维护源文档与 SPEC 的同步（v1.0.1 docs-only patch 是第一次练习）。
- 开发团队必须养成「先查 SPEC」的习惯。

**技术债务**：FU-3 / FU-4 从 SPEC 推迟到 v1.1+，现已登记在 `../../03_Engineering/KNOWN_ISSUES.md`。

## 是否允许未来修改

**极低可逆性**——这是产品元规则。若需要修改，必须通过 Project Lead 显式确认并新增一条 ADR 记录（如 ADR-NNN 声明新 SoT）。
