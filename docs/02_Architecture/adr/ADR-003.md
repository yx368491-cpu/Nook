# ADR-003 · 永不放 Web Push / 系统通知 / Email 推送

## Status

✅ **Accepted**

## Date

2026-06-27

## 修正历史

- v1.0.1 docs-only patch（FU-1 sync）— 从 MUST 降为 NEVER N0

## Context

`../../01_Product/Nook-PRODUCT.md` § 4 M12 把「Web Push Notification（你被想念了）」列为 MUST。但产品定位为「零噪音信息获取权」——所有推送都是 IG 和 Twitter 的「被迫打搅」功能；Nook 应依赖用户主动打开 + 应用内未读。

`../../01_Product/Nook-INTERVIEW-spec.md` § 2.6 NOTIF-2 + Round-3 Q4 也明确「无 Web Push」。

## Decision

v1.0 永不放 Web Push / 系统通知 / Email 推送。所有「在 / 不在场」提示仅走：
- F-NOTIF-01 应用内未读小红点。
- F-ST-02 Tab title `[N] Nook_v1.0`。

影响：
- 删 send-push Edge Function。
- 删 `../../01_Product/Nook-PRODUCT.md` § 4 M12 → 迁入 NEVER N0。
- 架构层硬不实现（无对应 CAP，无 Service Worker 推送通道）。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **应用内未读 + Tab title** | 零外部依赖；无隐私风险 | 用户不在浏览器时无法感知新消息 |
| **Web Push 轻量通知** | 可触达离线用户 | 违背「不受打搅」的产品哲学；国内 FCM 不可用 |

## Consequences

- 产品哲学「不受打搅」得到架构层的强制执行。
- 用户不能在浏览器外感知新消息——这是设计意图。
- 减少供应商依赖（不需要 FCM / APNs / Email 服务）。

## 是否允许未来修改

**不可逆**——Never-Do 边界（SPEC § 1.7.2）。任何提议必须先过 `../../01_Product/Nook-PRODUCT.md § 2` 反模式清单。
