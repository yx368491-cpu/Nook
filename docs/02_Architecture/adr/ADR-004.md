# ADR-004 · F-MSG-07 列级软隐藏（不物理 DELETE）

## Status

✅ **Accepted**

## Date

2026-06-27

## 修正历史

- 初始设计为物理 DELETE → Stage 7 评审改为列级软隐藏

## Context

`../../01_Product/Nook-SPEC.md` AC.10 明确规定：「仅自己端消失，其他朋友端仍可见」。原始设计使用物理 DELETE（`DELETE FROM messages WHERE id=X AND sender_id=auth.uid()`）会导致 DB row 消失，其他朋友端无法读取该消息 —— 违反 AC.10。

## Decision

F-MSG-07（本地删除）实现为：

- DB 列 `messages.deleted_by_sender_at timestamptz NULL`。
- 列级 GRANT：`revoke update on messages from authenticated; grant update (body, deleted_by_sender_at) on messages to authenticated;`。
- 客户端 UI：sender 看到「已删除」占位；其他朋友仍看原 row。
- 30 天 pg_cron 清理时，`deleted_by_sender_at` 非空的消息也被物理删除（与所有消息一视同仁）。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **物理 DELETE（原始设计）** | 存储省；实现简单 | 违反 AC.10；其他端不可见 |
| **列级软隐藏（选中）** | 符合 AC.10；RLS 天然 | 多一列；30 天后仍需清理 |
| **独立 deleted_messages 表** | 隔离干净 | 额外表；查询复杂度上升 |

## Consequences

- messages 表新增一列，不增加表数。
- EF `admin-delete-friend` 不触发此列（admin 删除是 left_at）。
- 其他朋友端不受 sender 删除影响。

## 是否允许未来修改

**中可逆性**——可加 `deleted_by_all_at` 升级为「全员删除」，但要改 RLS policy。
