# ADR-012 · 认证方案 Supabase Auth + 自建 invite-token

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

Nook 是邀请制系统（unique入口），不允许陌生注册。认证方案需要满足：email + password 登录、一次性 invite token、Owner-only admin 密码重置、不发验证邮件。

## Decision

**Supabase Auth（GoTrue）+ 自建 invite-token（24h 一次性）**。

- Owner 注册：`supabase.auth.signUp()` → EF `admin-bootstrap` 设 role='owner'。
- Friend 注册：EF `friend-signup` 一站式（验证 invite → signUp → 写 profile → 建 1:1 conv → 标 used_at）。
- 登录：`supabase.auth.signInWithPassword()`（标准流程）。
- 密码重置：EF `admin-reset-password` → `supabase.auth.admin.updateUserById()`（不发邮件，由 Owner 手动重置）。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Supabase Auth + Magic Link** | 无密码体验 | 国内邮件送达不稳 |
| **Supabase Auth + OAuth（Google/GitHub）** | 社交登录方便 | 稀释 Invite-only 入口唯一性 |
| **Supabase Auth + invite-token（选中）** | 邀请制哲学；不依赖邮件 | Friend 注册需要 EF |

## Consequences

- Service Role Key 仅 EF 持有，永不进客户端。
- Friend 注册走 EF 而非直接 `signUp`，增加 100ms 延迟但保障原子性。
- 密码重置不走邮件——Owner 主动管理。

## 是否允许未来修改

**高可逆性**——可加 Passkey（WebAuthn）作为 v1.1+ 选项，但 invite-token 入口设计必须保留。
