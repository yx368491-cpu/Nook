# ADR-005 · unread_counts 基于 last_read_at 游标

## Status

✅ **Accepted**

## Date

2026-06-27

## Context

原始设计考虑使用「today 24h 滑动窗口」来计算未读计数。这种方式存在根本性问题：如果用户读了历史消息（24h 前），窗口仍然会把它们算作未读；相反，如果用户 24h 没打开 Nook，窗口会错过真实未读。

## Decision

未读计数采用 **cursor-based 游标**（基于 `conversation_members.last_read_at`）。

- 新列 `conversation_members.last_read_at timestamptz default now()`。
- RPC `fn_mark_conversation_read(p_conv uuid)`：客户端进入会话时调用。
- RPC `fn_unread_counts(p_user uuid)`：使用游标计算 `count(messages.created_at > cm.last_read_at)`。
- 游标粒度 = per-conversation，而非 per-user-global。

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| **24h 滑动窗口** | 实现简单；无额外列 | 读历史 vs 读新消息 语义错误 |
| **last_read_at 游标（选中）** | 真实未读；粒度精确 | 需要维护游标列；多一个 RPC |

## Consequences

- conversation_members 表多一列 `last_read_at`。
- 客户端必须在会话可见时调用 `fn_mark_conversation_read`。
- `fn_unread_counts` 由 TanStack Query 30s 轮询 + Realtime 推送增量。

## 是否允许未来修改

**中可逆性**——改 algorithm 即可（如从游标改为时间戳比较），但游标语义已是业界标准。
